## 1. 核心原则

| 原则 | 一句话说明 |
| --- | --- |
| 配置外置 | 所有可调参数只在 `config/` 定义；实现文件**只读**配置，不得二次声明默认值。 |
| 校验外置 | 校验下沉到同名 `X_checks.py` 伴随文件；实现文件保留核心逻辑的可读性，不被校验淹没。 |
| 文档同步 | 每个实现文件有文件头；每次增删文件同步更新 `Doc/Index.md`。 |
| 简洁优先 | 完整实现功能的前提下，代码越短越好；反对冗长、反对复制粘贴。 |
| 表达优于循环 | 优先向量化 / 推导式 / 高阶函数；`for` 仅在确无更清晰写法时使用。 |
| 中文注释 | 注释、docstring 一律中文，解释「为什么」而非复述「做什么」。 |

## 2. 目录职责

| 目录 | 职责 | 是否含可配置参数 |
| --- | --- | --- |
| `config/` | 全部参数 + 配置加载与校验 | **是（唯一来源）** |
| `data/` | 数据读取、预处理、Dataset/DataLoader | 否（从 config 读） |
| `model/` | 网络结构定义 | 否 |
| `train/` | 训练 / 优化 / 评估循环 | 否 |
| `clone_loop/` | 行为克隆闭环（采样—推理—回灌） | 否 |
| `vis/` | 可视化、日志渲染 | 否 |
| `Doc/` | 文档与索引 | — |

> 判断一个值该不该进 `config/`：**如果换个实验/环境可能要改它，就进 config。** 形状推导、常量级别的数学定义（如 `math.pi`）不算配置。

## 3. 逐文件文档（文件头）

每个 `.py` 实现文件**必须**以模块 docstring 开头，模板如下：

```python
"""<一句话职责：本文件是做什么的>

模块: train/loop.py
依赖: model.policy, data.loader, config
读取配置: train.lr, train.epochs, train.device   # 本文件用到的 config 键，逐项列出
对外接口:
    - train_one_epoch(model, loader, cfg) -> float   # 返回平均 loss
    - evaluate(model, loader, cfg) -> dict
说明: <可选，记录关键设计取舍 / 注意事项>
"""
```

要求：

- **读取配置** 一栏必须如实列出本文件用到的所有 config 键，便于审查「是否有未走 config 的硬编码」。
- **对外接口** 只列公开函数/类（下划线开头的私有项不必列），一行一个，标注返回。
- 文件头与真实代码不符视为缺陷，等同代码 bug。

**豁免**：校验伴随文件 `X_checks.py` **不需要**文件头——它附属于 `X.py`、职责自明（就是 `X.py` 的校验）。文件内每个 `validate_*` / `check_*` 仍按 §7.2 标注校验对象即可。

## 4. 中文注释规范

- **语言**：注释、docstring 全部中文；标识符（变量、函数名）仍用英文。
- **函数 docstring**：公开函数必须写，私有/一目了然的可省。模板：

```python
def sample_actions(obs, policy, cfg):
    """根据观测采样动作。

    参数:
        obs:    形状 (B, C, H, W) 的图像观测张量
        policy: 已加载权重的策略网络
        cfg:    配置对象，读取 cfg.clone.temperature
    返回:
        (B, action_dim) 的动作张量
    """
```

- **行内注释**：只在「为什么这么写」不显然时出现。复述代码的注释（`i += 1  # i 加一`）禁止。
- **TODO / FIXME**：统一格式 `# TODO(姓名): 内容`，便于检索。

## 6. 配置集中管理

### 6.1 结构

```
config/
├── default.yaml      # 全部参数的默认值（唯一数据来源）
├── schema.py         # 参数的类型定义 + 校验（dataclass / pydantic）
└── __init__.py       # load_config() 入口：读 yaml → 构造 schema → 返回配置对象
```

- 参数值写在 `default.yaml`；类型与约束写在 `schema.py`；二者分离，值与校验各自单一来源。
- 环境差异用 `config/<env>.yaml` 覆盖默认，不在实现文件里分支判断。

### 6.2 铁律

1. **实现文件不得出现字面量配置**。即：实现文件里不允许 `lr = 3e-4`、`batch_size = 32`、`epochs = 100` 这类「换实验会改」的赋值。
2. 实现文件通过参数接收配置对象 `cfg`，或 `from config import load_config` 读取，**只读不改**。
3. 同一参数只允许在 `default.yaml` 出现一次；禁止在多个文件各写一份默认值。

### 6.3 正反例

```python
# ✅ 正确：值来自 config
def build_optimizer(model, cfg):
    return torch.optim.AdamW(model.parameters(), lr=cfg.train.lr)

# ❌ 错误：实现文件里二次配置
def build_optimizer(model):
    lr = 3e-4            # 配置泄漏到实现文件，禁止
    return torch.optim.AdamW(model.parameters(), lr=lr)
```

## 7. 校验规范

目标：**只保留必要校验，既不污染实现文件的核心逻辑，也不堆成一个巨型校验文件。**

### 7.1 校验逐文件分离、就近伴随

不设全局校验大文件（后期会膨胀到几十万行、可读性归零）。规则：

> **每个实现文件 `X.py` 配一个同目录的伴随文件 `X_checks.py`。**
> `X.py` 的校验逻辑只写在 `X_checks.py` 里；`X.py` 本体只在入口处调用一次，保持核心逻辑干净。

```
train/
├── loop.py            # 实现：只剩核心逻辑
└── loop_checks.py     # 校验：validate_*、check_* 函数，随 loop.py 一起增删
```

- 伴随文件随实现文件**一对一**存在、同生共死；删实现文件就删它的 `_checks.py`。
- 每个 `_checks.py` 只服务它对应的那一个实现文件，因此天然短小、好读。
- 实现文件通过**一次调用或装饰器**接入校验，不把断言散落进函数体：

```python
# train/loop.py —— 本体干净，校验下沉到伴随文件
from train.loop_checks import check_train_inputs

def train_one_epoch(model, loader, cfg):
    check_train_inputs(model, loader, cfg)   # 入口处一次性校验
    ...                                      # 以下全是核心逻辑
```

唯一例外：**配置参数**的合法性校验仍写在 `config/schema.py`（它本就是配置的单一来源，且在加载期一次拦截，天然有界，不会无限膨胀）。

实现函数体内尽量只剩核心逻辑。确需内联的断言，应少而精。

### 7.2 校验必须标注对象

每处校验必须说明它**针对谁**（哪个函数 / 变量 / 配置项）：

```python
# 校验对象: cfg.train.lr —— 学习率必须为正
assert cfg.train.lr > 0, "train.lr 必须 > 0"

# 校验对象: 函数 sample_actions 的入参 obs —— 必须是 4 维图像张量
def _check_obs(obs):
    assert obs.ndim == 4, "obs 期望 (B,C,H,W) 四维"
```

未标注对象的校验视为不合格。

### 7.3 必要性

- 能由 `schema.py` 在加载期拦截的，不要在运行期 `_checks.py` 里重复校验。
- 同一约束只校验一次，靠近来源（配置在加载处，数据/入参在实现文件的入口调用处）。
- 不写「不可能为真」的防御代码；保持核心路径干净。

## 8. 代码简洁与结构

- **单一职责**：一个函数只做一件事；超过约 40 行先考虑拆分。
- **不重复**：第二次复制同段逻辑时，抽成函数。
- **早返回**：用 guard-clause 减少嵌套，避免深层 `if`。
- **命名即注释**：好名字能消掉的注释就别写注释。
- **删除死代码**：注释掉的旧代码不进仓库，交给 git 记忆。
- **导入**：标准库 / 第三方 / 本项目 三组，组间空行。

## 9. 少用 for 循环

完整功能优先，但在等价且不更晦涩时，按以下顺序选择：

| 优先级 | 写法 | 适用 |
| --- | --- | --- |
| 1 | 向量化（numpy / torch / pandas） | 数值批量运算 —— 本项目主战场 |
| 2 | 推导式 / 生成器表达式 | 构造列表、字典、过滤映射 |
| 3 | `map` / `filter` / `itertools` / `sum` 等内建 | 逐元素变换、聚合 |
| 4 | `for` 循环 | 含副作用、提前中断、或上述写法反而更难读时 |

```python
# ✅ 向量化
loss = ((pred - target) ** 2).mean()

# ✅ 推导式
paths = [p for p in root.iterdir() if p.suffix == ".png"]

# ❌ 能向量化却手写循环
total = 0.0
for i in range(len(pred)):        # 既慢又长
    total += (pred[i] - target[i]) ** 2
```

> 原则不是「禁止 for」，而是「当存在更清晰的表达时，不要用 for」。可读性优先于教条。
